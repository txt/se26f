# Week 6 — exit=fn and Full State Schema

## The Bear

Uncomment `-- S6` exit blocks.

```
-- seed 42
Bear#1 foraging.   hunger=80 energy=60
>> leaving foraging: burned 10 cal, reset found_food
Bear#1 sniffs.     hunger=80 energy=50
```

The bear now cleans up when leaving a state.
Exit fires *before* entering the next state.
The engine did not change — a new field unlocked new behaviour.

## Tests T19–T21

    T19 exit fires before action of next state
    T20 exit receives (state_name, payload) — both correct
    T21 state with no exit= field runs without error

```lua
local log = {}

local rules = {
  a = {
    action = function(p) log[#log+1]="action_a" end,
    exit   = function(s,p)
               log[#log+1]="exit_"..s
               p.visited_a=true end,
    transitions = {go="b"}
  },
  b = {
    action = function(p) log[#log+1]="action_b" end,
    transitions = {}
  }
}
local agent = {start="a", current="a",
               queue={"go"}, states=rules}
M.start(agent)

-- T19: exit_a fires before action_b
local ea = 0; local ab = 0
for i,v in ipairs(log) do
  if v=="exit_a"   then ea=i end
  if v=="action_b" then ab=i end end
assert(ea < ab)

-- T20: exit received state name "a"
assert(agent.visited_a == true)

-- T21: state b has no exit, ran cleanly
assert(log[#log] == "action_b")
```

## TLAs

| # | TLA | Expansion | Definition | Bear example |
|---|-----|-----------|------------|--------------|
| 16 | OCP | Open/Closed Principle (state level) | A state table is open for new fields (exit, on_error) without modifying the engine. | Adding `exit=fn` to `foraging` required zero engine changes. The engine pattern-matches known fields. |
| 17 | TDP | Table-Driven Programming | Encode behaviour in data tables rather than code branches. The interpreter is simple; the data is rich. | The entire bear machine is a Lua table. The engine is 5 lines. All complexity is in the table. |
| 18 | PLS | Principle of Least Surprise | Code should behave the way a reasonable reader expects. Surprises are bugs waiting to happen. | `exit` return value is explicitly ignored. If it were used as the next event, control flow would be invisible. |

## Anti-Patterns

| # | Name | Symptom | Consequence | Bear smell |
|---|------|---------|-------------|------------|
| 16 | Implicit Control Flow | Exit function return value secretly controls next state | Invisible transitions; reader must trace function to understand machine | `exit=fn` where `fn` returns a state name and the engine uses it without documentation |
| 17 | Fat State | Every state carries 15 fields; most are nil | Cognitive load; easy to accidentally use nil field | Requiring `on_enter`, `on_exit`, `on_error`, `on_timeout` even when all are nil |
| 18 | Shotgun Surgery | Adding one feature requires editing every state in the machine | High coupling; easy to miss one | If adding `trace=true` to each state required editing all 5 states manually |

## Python Idioms

| # | Idiom | Bad (newbie) | Good (idiomatic) |
|---|-------|--------------|-----------------|
| 16 | `dict.get` with default | `if "exit" in state: state["exit"](s,p)` | `exit_fn = state.get("exit"); if exit_fn: exit_fn(s,p)` — one lookup, clear intent |
| 17 | `match`/`case` on types | `if isinstance(t,str): ...; elif callable(t): ...` | `match transition: case str(): ...; case _ if callable(_): ...` — structural pattern matching (3.10+) |
| 18 | `dataclass` `field(default=None)` | Manually check every optional field for existence | `@dataclass class State: exit: Callable=None; guard: list=None` — None is the documented absent-value |
